# Building an IDA Pro Processor Module + Loader for Qualcomm Adreno "AFUC" Command-Processor Microcode (a5xx-a7xx)

## TL;DR
- **The single best path is to port Mesa's `afuc` directly:** treat `src/freedreno/afuc/afuc.xml` (an isaspec/rnn XML description of the 32-bit ISA) as ground truth, and either hand-write IDA instruction tables from it or auto-generate them by parsing the XML. The ISA, firmware container formats, and a full assembler/disassembler/emulator are all open source in Mesa, so almost nothing needs original reverse engineering.
- **There are three distinct things to model, do not conflate them:** (1) the **AFUC** microcontroller in the CP (PFP+ME on a3xx-a5xx; unified **SQE** on a6xx; split **BV/BR** SQEs on a7xx) - this is your processor module target; (2) the **GMU** firmware, which is a *separate* Cortex-M-class core (use IDA's stock ARM module, NOT afuc); (3) the **ZAP** shader (signed PIL/MDT blob, mostly afuc + an embedded ir3 shader).
- **Recommended architecture:** an IDA loader (`accept_file`/`load_file`) that strips the SQE/PFP header, sets `instruction base 0x1000`-style segmentation, and selects the processor by GPU generation; plus a generation-parameterized processor module (`processor_t` with `ev_ana_insn`/`ev_emu_insn`/`ev_out_insn`). Build for IDA 9.0's SDK (the processor API changed at 9.0; `ida_struct`/`ida_enum` were removed in favor of `ida_typeinf`).

## Key Findings

### What AFUC is
Adreno's command processor runs custom Qualcomm microcontroller(s) executing a fixed 32-bit instruction set Mesa calls **afuc** - recently renamed **QRisc** in Mesa's tree (Mesa 26.1.0 Release Notes, dated 2026-05-06, list "freedreno: Rename afuc to QRisc"; a follow-up commit "freedreno: fix a few missed afuc -> qrisc renames" confirms the in-tree rename). Per Chester Lam's reporting in Chips and Cheese ("Correction on Qualcomm iGPUs"), corroborated by a Mesa/Turnip developer: "Adreno's command processor actually consists of microcontroller(s) running a MIPS-inspired instruction set starting from the 5xx generation. Even though 5xx's command processor moved away [from] the instruction set used by AMD's R600, it retained..." a similar ME+PFP arrangement. Lineage matters: Adreno IP descends from AMD's Imageon handheld graphics division, which **AMD sold to Qualcomm in January 2009 for $65 million in cash** (per AMD's SEC Form 8-K: "the sale of technology assets, intellectual property and resources that formed the basis of its Handheld business to Qualcomm for $65 million in cash"). That is why the CP uses AMD-style **PFP (Prefetch Parser) + ME (Micro Engine, a.k.a. "PM4 firmware")** naming and a PM4-like packet format.

Generational CP structure:
- **a3xx/a4xx/a5xx:** two microcontrollers - **PFP** (`aXXX_pfp.fw`) and **ME/PM4** (`aXXX_pm4.fw`). Only the PFP reads the command stream and passes commands to ME via a queue.
- **a6xx:** per Chester Lam, "Adreno 6xx unifies the ME and PFP into a single 'SQE' microcontroller, and adds a separate LPAC microcontroller to handle low priority compute" (`aXXX_sqe.fw`).
- **a7xx:** the SQE is split into **BV** (binning visibility) and **BR** (render) microcontrollers - a "dual-core" CP - to overlap binning and tile rendering. Driver thread-control commands gate which core executes which packets.

### The AFUC ISA (from `afuc.xml`)
- **Instruction word:** fixed 32-bit (`<bitset name="#instruction" size="32">`, doc: "All instructions are 32b").
- **Opcode location is family-dependent** (this is the single most important implementation gotcha): register/ALU 2-src form carries its 5-bit opcode in **bits [0:4]**; ALU *immediate* forms carry it in **bits [27:31]**; control-flow/branch instructions use a 6-bit opcode in **bits [26:31]**.
- **Register file:** GPRs `$00`-`$1b` selected by 5-bit `#src`/`#dst` fields. Values 0x1c-0x1f are special registers, and the src/dst meanings differ (confirmed verbatim in `afuc.xml` enums): `#specialsrc` = {0x1c `$rem`, 0x1d `$memdata`, 0x1e `$regdata`, 0x1f `$data`}; `#specialdst` = {0x1c `$rem`, 0x1d `$addr`, 0x1e `$usraddr`, 0x1f `$data`}. There are named aliases used by compiled code: `$lr` (link register), `$sp` (stack pointer). `$data` reads/writes the command-stream FIFO; `$addr`/`$memdata` form an indirect memory-access pair; `$regdata` reads GPU registers.
- **Modifiers:** `(rep)` repeat = bit 26; `(xmovN)` (N=0..3) executes extra movs based on `$rem` (2-bit `#xmov` enum). a7xx adds an `(sdsN)` modifier on cwrite and a `(peek)` modifier.
- **ALU ops:** add, addhi, sub, subhi, and, or, xor, not, shl, ushr, ishr, rot, min, max, mul8, msb, cmp, plus a7xx-only bic, setbit/clrbit, ubfx (unsigned bitfield extract), bfi (bitfield insert). 2-src reg form: DST=[16:20], SRC1=[21:25], SRC2 in low bits; immediate-12 form: RIMMED=[0:11], DST=[16:20], SRC1=[21:25], opcode[27:31]; 1-src immediate form: IMMED=[0:15], DST=[16:20].
- **Control register access:** `cwrite SRC, [OFFSET + BASE]` (opcode [27:31]=`10110`) and `cread DST, [OFFSET + BASE]`. BASE/IMMED=[0:11], DST=[16:20], with a `(sds)` field and pre-increment addressing flag. Control registers are referenced symbolically (e.g. `@IB_LEVEL`, `@REG_WRITE_ADDR`, `@REG_READ_ADDR`, `@LOAD_STORE_HI`).
- **Memory access:** `load DST, [OFFSET + imm]` and `store` with pre-increment addressing.
- **Control flow:** conditional branches `brne`/`breq` (immediate form opcode [26:31]=`110000`/`110001`; bit-test form `brne SRC, bN, #label`); the pre-a7xx **`jump #label` is an alias for `brne $00, b0, #label`** (no dedicated opcode). `call`/`ret` use a CP call stack; `waitin` (opcode `110110`) waits for the next packet, with the delay-slot convention `mov $01, $data` to load the next packet header. `setsecure` (opcode `111011`) toggles secure mode. `iret` returns from interrupt; `preempt`/`preemptleave` handle preemption. **AFUC has branch delay slots** (the test asm comments "there's a branch delay slot, so this isn't executed yet") - your module must set `PR_DELAYED` semantics / handle the delay slot in `emu`.
- **a7xx "new-style branches":** an a7xx-only `#jump-a7xx` family (opcode [26:31]=`110111`) adds `jumpr` (indirect jump, `jump $lr` used as a return paired with `bl`) and `sret` (compiled-code return that restores the stack frame). `bl` ("jump to target and store return address in `$lr`") supports a proper call/return convention in a7xx compiled firmware.
- **Generation gating:** `afuc.xml` uses `<gen min=".." max="..">` on bitsets (gen 5=a5xx, 6=a6xx, 7=a7xx). Several mnemonics have 2-3 encodings differing only by `<gen>`, and **several opcode values changed in a7xx** (e.g. `shl` 01001→10010, `ishr` 01011→10100, `rot` 01100→10101, `cmp` immed-10000→reg-01101). Even `nop` differs: a5xx nop is all-zeros, but a6xx+ made all-zeros illegal and nop sets bit 25. This is why a single decoder must be parameterized by GPU generation, exactly as `afuc-asm -g 6 / -g 7` is.

### Mesa `afuc` source layout (what to port/reference)
The directory `src/freedreno/afuc/` (originally landed by Rob Clark in MR !6070, "freedreno: slurp in afuc") contains:
- **`afuc.xml`** - the isaspec ISA description (primary source of truth for encodings).
- **`disasm.c`** - disassembler driver (built as `afuc-disasm`); reads firmware, prints the version header, sets "instruction base 0x1000," disassembles.
- **`asm.c`** + **`parser.y`**/**`lexer.l`** (`asm.h`) - assembler (`afuc-asm`); supports labels, `.align`, multiple sections, relocations, label-as-immediate.
- **`emu.c`**/`emu.h` (+ register-support files) - an interactive emulator (added in MR !10944) that models GPRs, control registers, GPU registers and memory; can run the bootstrap routine to populate the packet table. Useful for recovering semantics and for understanding the packet-dispatch jump table.
- **`util.c`**, headers defining `struct afuc_instr`, register names, control-register tables.
- **`meson.build`** - builds the tools and a regression test (`afuc_test.asm` → `.fw`, round-tripped). The reference asm at `src/freedreno/.gitlab-ci/reference/afuc_test.asm` is an excellent, public, legally-clean test corpus showing every instruction in use (Euclid's algorithm, a CP_MEMCPY via load/store, secure-mode toggling, etc.).
- isaspec generator infrastructure lives under `src/compiler/isaspec/` (the `encode.py`/decode generator, `isaspec-decode.h`); `docs.mesa3d.org/isaspec.html` documents the XML schema (bitsets, overrides, expressions, `<gen>`, `branch`/`absbranch` field types for branch-label printing).

### Firmware container formats and how to get to raw instructions
- **PFP/PM4/SQE `.fw` files** are the raw afuc instruction stream with a small header of version words. The kernel (`drivers/gpu/drm/msm/adreno/a6xx_gpu.c`) computes the SQE version as: `version = data[0]; if ((version & 0xf) != 0xa) return version; else version = (version & ~0xfff) | ((data[2] & 0xfff000) >> 12)` - and the comment notes "compared to kgsl, we've already stripped off the first dword." So **KGSL/on-device blobs have one extra leading dword that the upstream linux-firmware copies have stripped.** Mesa's disasm prints "instruction base: 0x1000" and a "Version:" word (e.g. a630_sqe.fw version `016ee17a`). The first one or two 32-bit words are header/version, then the instruction stream begins; afuc's disasm shows the first words as `[01000001]`/`[01000000]` bracketed "nop"-like header words.
- **Security/version checks** in `a6xx_ucode_check_version()`: a630 needs ≥0x190 (1.90), a650 needs ≥0x195 (later ≥0x095 in newer tree), a660/a7xx are considered safe. The `(buf[0] & 0xf)==0xa` path indicates the newer version-encoding scheme. The `WHERE_AM_I` opcode capability gates `has_whereami`.
- **GMU firmware** (`aXXX_gmu.bin`) is a **separate Cortex-M-like core - NOT afuc.** Disassemble it as ARM Thumb, not with this module.
- **ZAP shader** (`aXXX_zap.{mdt,b00,b01,...}` or `.mbn`) is a Qualcomm **PIL/MDT split-binary**: an ELF-like `.mdt` header/metadata plus `.bNN` segment files, signed for the TrustZone "secure world." Per Rob Clark, the a6xx zap fw "is just normal AFUC instructions with an embedded ir3 shader, but no packet-table," so it can partly be parsed by afuc too. PIL MDT = ELF header + program headers; concatenate segments per the program headers to reconstruct, or use `pil-squasher`.
- **Sample sources:** kernel `linux-firmware/qcom/` (a300_pfp.fw, a300_pm4.fw, a420_pfp.fw, a530_pfp.fw, a530_pm4.fw, a630_sqe.fw, a650_sqe.fw, a660_sqe.fw, a702_sqe.fw, leia_pfp_470.fw/leia_pm4_470.fw, yamato_pfp.fw/yamato_pm4.fw, etc.); Debian `firmware-qcom-soc`; on-device `/lib*/firmware/a*_sqe.fw` and `/vendor/firmware/`. Newer a7xx use names like `gen70900_sqe.fw` (Galaxy S24). Naming maps to `adreno_device.c`/`a6xx_catalog.c` `.fw[ADRENO_FW_*]` tables.

### Relationship to PM4 packets
The SQE/PFP firmware is precisely what parses and executes the PM4 command stream. Packet headers: type-4 (`CP_TYPE4_PKT`, register writes) and type-7 (`CP_TYPE7_PKT`, opcode + count, with parity bits - see `OUT_PKT7`: bit15 = parity(count), bits[16:22]=opcode, bit23=parity(opcode)). The microcode contains a **packet/jump table** indexed by CP opcode; each `CP_*` handler (e.g. `CP_ME_INIT`, `CP_MEM_WRITE`, `CP_SET_DRAW_STATE`, `CP_INDIRECT_BUFFER`, `CP_SMMU_TABLE_UPDATE`) is an afuc routine. Mesa's `adreno_pm4.xml` documents `CP_*` opcodes and packet layouts; `adreno_control_regs.xml` documents control registers like `@IB_LEVEL`.

The CVE-2025-21479 work shows the practical payoff. Per zhuowei (github.com/zhuowei/cheese), diffing Galaxy S24 `gen70900_sqe.fw` v675 (April, build S921USQU4BYD9) vs v676 (May, build S921USQS4BYE4) showed exactly one type of change: `0164: 2a440003 and $04, $12, 0x3` → `0164: 2a440007 and $04, $12, 0x7` - "Every access to `$12` now ANDs with `0x7` instead of `0x3`. There are no other changes." (Corroborated by xploitbengineer: "In the April firmware, the `$12` register is masked with the constant 0x3. In the May patch, this was changed to 0x7.") Register `$12` is the IB level; a7xx added IB3, which made the SDS (set-draw-state) buffer alias the kernel ring buffer under the old `&0x3` mask, allowing privileged packets such as `CP_SMMU_TABLE_UPDATE` to execute from a userspace-controlled buffer.

### Prior art (don't reinvent)
- **Mesa `afuc` itself** is the de-facto disassembler/assembler/emulator; most researchers (including the CVE-2025-21479 authors) just use `afuc-disasm`/`afuc-asm`.
- No mature, public **IDA processor module for afuc** appears to exist yet - this is a genuine gap, making the project worthwhile.
- **Closely analogous IDA processor-module projects to learn from:** `programa-stic/hexag00n` (Qualcomm Hexagon, Python, with the best write-up of the ana/emu/out cycle), `mandiant/idawasm` (loader + processor module pair, good template for the loader side), the Hex-Rays "Scriptable Processor modules" blog, the IDA SDK `module/script/proctemplate.py` and `msp430.py` samples, Ninji's "Mouse Adventures #7" (jump-table autodetection via `add_cref`, exactly what you need for the afuc packet table), Quarkslab's and delroth's processor-module write-ups, and Apriorit's Xtensa `ev_ana_insn` example.
- **Ghidra alternative:** SLEIGH. Relevant exemplars: `pietroborrello/ghidra-atom-microcode` (Intel microcode - a near-perfect analog of "GPU microcode in Ghidra," including the trick of packing odd-width instructions into addressable units), `oberoisecurity/ghidra-processor-module-generator` (auto-generates SLEIGH from instruction lists), the iRISC syndicate's NIC-firmware module, and the PT SWARM SLEIGH tutorial. A SLEIGH module gives you a decompiler for free.

## Details

### Recommended implementation plan for the IDA module + loader

**1. Generate, don't hand-maintain, the opcode tables.** Write a small Python script that parses `afuc.xml` and emits the IDA `instruc[]` table and a decoder. The isaspec XML already encodes every field position, pattern, `<gen>` gate, and display string. Two viable strategies:
- **(a) Build-time codegen:** parse `afuc.xml` → emit a Python dict of `{itype: (mnemonic, [field specs], gen_mask, features)}`. This is the most maintainable and tracks Mesa upstream as the ISA evolves. Reuse Mesa's own `isaspec` Python (`encode.py`) as a reference for how patterns/overrides/derived fields resolve.
- **(b) Link against libafuc:** compile Mesa's generated decoder (`isaspec-decode.h` + the afuc decode tables) into a small C helper that returns decoded instructions, and have the IDA module call it. Lower maintenance but adds a native build dependency. For an IDA Python module, (a) is cleaner.

Hand-writing tables is feasible (the ISA is small - a few dozen mnemonics) but you would duplicate the `<gen>` matrix and the family-dependent opcode-field logic by hand, which is exactly where bugs hide.

**2. Handle generations with one parameterized module.** Store the GPU generation (5/6/7) as a module/netnode setting chosen by the loader. In `ev_ana_insn`, select the encoding variant by gen. This mirrors `afuc-asm -g N`. Concretely: keep three decode tables (or one table with per-entry gen-min/gen-max) and switch on the segment's gen.

**3. Processor module skeleton (`processor_t` subclass, `PROCESSOR_ENTRY`):**
- `ev_ana_insn(insn)`: read one 32-bit word (AFUC's "word" is 4 bytes, so use 32-bit reads / `ua_next_long`-style), match by family (test [26:31] for branches/control first, then [27:31]/[0:4] for ALU), fill `insn.itype`, operands (`o_reg` for GPRs/special regs, `o_imm` for immediates, `o_near` for branch targets, a custom `o_idpspec`/`o_displ` for `[OFFSET + BASE]` control/memory operands), set `insn.size = 4`. Return 4 or 0.
- `ev_emu_insn(insn)`: add flow refs (`add_cref(... fl_F)`), branch/call refs (`fl_JN`/`fl_CN`), and - crucially - handle the **branch delay slot** (the instruction after `jump`/`call`/`waitin`/`brne` still executes; add a flow ref to it). Implement **packet-table / jump-table recovery** à la Ninji: detect the dispatch pattern at bootstrap, then create code refs to each `CP_*` handler and name them from `adreno_pm4.xml` opcode names. Mark `ret`/`iret`/`sret`/`jumpr $lr` as returns; mark unconditional `jump`/infinite-loop error stubs with `CF_STOP` where appropriate.
- `ev_out_insn`/`ev_out_mnem`/`ev_out_operand`: render mnemonic + operands; use `out_value` for immediates so users can re-base them, print special regs by name (`$data`, `$addr`, `$rem`, `$lr`...), and print `(rep)`/`(xmovN)`/`(sds)` prefixes. Mirror afuc's display strings.
- Register set: define `$00..$1b` plus special regs and `$lr`/`$sp` as named registers in `reg_names`; provide fake code/data segment registers as IDA requires.
- Features: opcode `CF_*` flags (CF_USE1/CF_CHG1/CF_JUMP/CF_CALL/CF_STOP) per instruction. Use `is_switch`/`switch_info_t` for the packet jump table so IDA renders it as a switch.

**4. Loader module (`accept_file`/`load_file`):**
- `accept_file`: recognize SQE/PFP/PM4 blobs. Heuristics: small file (a few KB-tens of KB), known version-word pattern in word[0] (`(w0 & 0xf)==0xa` for new scheme, or plausible 1.xx in low 12 bits), and/or filename match (`*_sqe.fw`, `*_pfp.fw`, `*_pm4.fw`, `genNNNNN_sqe.fw`). Return a descriptive type string per detected generation.
- `load_file`: strip the header (1 dword if KGSL-style/on-device with the extra leading dword, else start at the documented offset), create a CODE segment at the instruction base (Mesa uses 0x1000; note afuc addresses are in 32-bit-word units in the disasm listing), set the processor type to your module, stash the GPU generation, and add an entry point at the reset/bootstrap vector. Optionally create a separate data segment for the memory space the emulator models.
- For ZAP/MDT: optionally a second loader path that parses the PIL ELF program headers, concatenates `.bNN` segments, and loads the afuc portion (skipping/annotating the embedded ir3 shader).

**5. IDA SDK version considerations.** Target IDA 9.0's SDK. The processor module API was reworked across 7.x→9.0: callbacks are `processor_t` notification events (`ev_ana_insn`, `ev_emu_insn`, `ev_out_insn`, `ev_out_operand`, `ev_out_mnem`); in IDAPython the event names differ slightly (e.g. `notify_ana` vs C++ `ev_ana_insn`). In 9.0 the `ida_struct` and `ida_enum` modules were **removed** - use `ida_typeinf` for any structs/enums you create (e.g. for control-register enums); see Hex-Rays' "Porting Guide from IDA 8.x to 9.0." For 8.x compatibility, wrap the few differing calls. Python modules are simplest for iteration (link a `.py` into `procs/`); C++ gives full API access if you need it.

### AFUC special registers and control registers (for annotation)
Special registers to name in the module: `$rem` (remaining count, drives `(xmovN)` and `(rep)`), `$data` (command-stream FIFO read/write), `$addr`/`$memdata` (indirect memory access pair), `$regdata`/`$usraddr` (GPU register / user address access), `$lr` (link register, a7xx), `$sp` (stack pointer, a7xx compiled code). Control registers (accessed via cwrite/cread, symbolic `@NAME`): `@IB_LEVEL` (current indirect-buffer level - the CVE-2025-21479 register `$12`), `@REG_WRITE_ADDR`/`@REG_WRITE`, `@REG_READ_ADDR`, `@LOAD_STORE_HI`, plus the broader control-register set in `adreno_control_regs.xml`. Register `$12` holding IB level is a useful default annotation.

### Testing corpus and workflow
1. Build Mesa's `afuc-disasm`/`afuc-asm` and use them as your oracle: for each test firmware, diff your IDA disassembly against `afuc-disasm -g N`.
2. Use the public, license-clean `afuc_test.asm`/`afuc_test.fw` regression pair first (no firmware hosting concerns).
3. Then test real blobs: a530_pm4.fw/a530_pfp.fw (a5xx), a630_sqe.fw / a650_sqe.fw / a660_sqe.fw (a6xx), a730/gen70900_sqe.fw (a7xx).
4. Use Mesa's `afuc` emulator to validate control-flow/semantics on the bootstrap routine and packet-table population.

## Recommendations

**Stage 1 - Bootstrap (1-2 days).** Build Mesa's afuc tools; collect firmware (linux-firmware + a couple of a7xx blobs). Confirm header layout and instruction base on each generation by comparing `afuc-disasm` output to raw bytes. Decide Python vs C++ (recommend IDAPython for IDA 9.0).

**Stage 2 - Decoder via codegen (3-5 days).** Write the `afuc.xml` → IDA-tables parser. Implement `ev_ana_insn` with family-dependent opcode dispatch and the `<gen>` matrix. Validate byte-exact against `afuc-disasm -g {5,6,7}` on the test corpus. *Threshold to proceed:* 100% mnemonic+operand match on `afuc_test.fw` and a630_sqe.fw.

**Stage 3 - Emulation/xrefs + loader (3-5 days).** Implement `ev_emu_insn` (flow, delay slots, call/ret, branch refs), the loader (`accept_file`/`load_file`, header strip, gen selection, entry point), and packet-table/jump-table recovery with `CP_*` naming from `adreno_pm4.xml`. *Threshold:* clean function graph for a6xx SQE with named packet handlers.

**Stage 4 - Polish + a7xx specifics (2-4 days).** Add `bl`/`jumpr`/`sret`/`$lr`/`$sp` call-convention handling, `(sds)`/`(peek)` modifiers, control-register `@NAME` annotation via `ida_typeinf` enums, and switch_info_t for the packet table. Add ZAP/MDT loader path if needed.

**When to choose Ghidra/SLEIGH instead:** if you want a decompiler with minimal effort, write a SLEIGH module (model it on `ghidra-atom-microcode`); the afuc ISA is small and SLEIGH gives p-code + decompilation. Use IDA if your workflow/tooling is IDA-centric or you need IDA's specific analysis. Doing both is reasonable; the `afuc.xml`-to-tables generator can target either backend.

**Borrow aggressively:** hexag00n (ana/emu/out structure), idawasm (loader+proc pairing), Ninji's jump-table autodetection, and Mesa's emulator for semantics. Track Mesa upstream (the ISA is still evolving - a7xx/a750 support and the QRisc rename are recent), so codegen beats hand-tables for longevity.

## Caveats
- **`afuc.xml` is the authority, and it is incomplete/evolving.** Some control-register enums and a few opcode values were still being filled in; the XML has TODO markers. A handful of base bitsets could not be confirmed verbatim during research (exact SRC2/XMOV bit positions in `#alu-2src`, the `#control`/store/load/call/ret/iret/preempt opcode patterns, the `(sds)` field bits, and the `#control-base`/`@IB_LEVEL` enum body). Fetch `afuc.xml` lines ~150-190 and ~580-840 directly before finalizing those encodings; treat the inferred opcode values (and/ushr/min/max/mul8) as needing verification against the file or `afuc-disasm` round-trips.
- **Generation differences are real and silent:** opcode values and even `nop` change between a5xx/a6xx/a7xx; a wrong `-g` will mis-decode without obvious errors. Always parameterize by gen.
- **Branch delay slots** will corrupt control-flow analysis if unhandled - budget for them explicitly.
- **a5xx vs a6xx/a7xx asymmetry:** a5xx has separate PFP and PM4 streams (two files, two entry models); a6xx+ is a unified SQE. a5xx emulator/disasm support in Mesa has historically lagged (the regression test notes "a5xx is still TODO").
- **Do not point this module at GMU firmware** (`*_gmu.bin`, Cortex-M) or the signed-but-non-afuc parts of ZAP - those are different cores.
- **On-device vs linux-firmware blobs differ by one leading dword** (KGSL keeps it; upstream strips it). Detect and handle both, or your instruction stream will be off by 4 bytes.
- Some claims here rest on secondary sources (Chips and Cheese for the BV/BR split and MIPS-inspired lineage; mailing-list patch fragments for the kernel version-check logic; the AMD 8-K for the Imageon sale). They are consistent across sources but worth confirming against current Mesa/kernel trees before relying on exact constants.
